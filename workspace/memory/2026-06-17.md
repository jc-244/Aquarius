## Post-compact: v1.2.3 wasn't actually live — vercel.json schema fix (9472db1)

Discovered after `/compact`: `aquarius-seven.vercel.app` was still serving
v1.2.2 from 2026-06-14 (~98 min after the v1.2.3 push), with `age: 196783`
and `x-vercel-cache: HIT`. Both v1.2.3 release commits (04939da and
6b761d9) had **failed Vercel schema validation BEFORE the build started**,
so no new artifact was ever produced — the edge just kept serving the last
good deploy with the age clock ticking up.

Root cause: `vercel.json` had `"public": true`, a legacy v2 field Vercel
quietly removed from the schema. Felix saw the actual error in the Vercel
dashboard popup ("`vercel.json` schema validation failed ... should NOT
have additional property `public`"). Without team access we couldn't read
the build log via `npx vercel inspect` (Vercel API tokens are scoped to
*teams the user is a member of*, not to deployment IDs — Felix's Hobby-tier
token returned `Can't find the deployment "dpl_..." under the context
"fly-m1ss-projects"`), but the popup was enough.

Fix `9472db1` removes the one line. Pushed → Vercel re-ran with the valid
schema → build succeeded → edge flipped to v1.2.3 (verified `v=1217` +
visible `v1.2.3` from cache-busted `curl`) in ~60 seconds.

### Diagnostic principle worth keeping

The combination **rising `age` header + `x-vercel-cache: HIT` + new commit
on `origin/main` + no new content visible** is NOT "CDN slow to propagate"
— it is **deploy never produced an artifact**. Always query GitHub
Deployments API (`gh api repos/<owner>/<repo>/deployments`) before chasing
propagation theories. With the deployment record + status in hand, even
without team access, you can see `state: failure` and the popup target URL
— the failure mode is then visible to the team owner in one screenshot.
